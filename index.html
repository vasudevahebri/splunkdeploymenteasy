How to use this studio
  • Step 1: load or build your components. These are Splunk hosts and agents such as search heads, indexers, deployment servers, heavy forwarders, and universal forwarders.
  • Step 2: define your data sources. These are the logs, events, or metrics the customer wants to onboard into Splunk.

Beginner-friendly mental model

Component = Splunk infrastructure or agent. Data source = the actual business data being collected. Example: a Universal Forwarder on a Windows server is a component; the Windows Security event log is the data source.

Component inventory

This is where you list the Splunk hosts and agents that make up the platform. These rows now drive diagrams, commands, validations, placement guidance, operational ownership, health checks, and recurring incident context.

What this section means

A component is a Splunk platform node or agent. Typical examples are a search head, indexer, deployment server, cluster manager, heavy forwarder, or universal forwarder.

How to use it
  • Create one row per Splunk host or agent. Include owner, criticality, monitoring state, and last health-check details for daily support.
  • Public IP is used for SSH and admin access in cloud builds.
  • Private IP is used for Splunk-to-Splunk traffic such as clustering, distributed search, and deployment traffic.

Data source inventory and onboarding inputs

This is where you describe what data the customer wants Splunk to collect. The app then turns those rows into placement guidance, onboarding steps, validation SPL, troubleshooting paths, exact guidance on where to update apps or add-ons, and daily operational source-health tracking.

What “data source” means

A data source is the log, event, metric, API feed, or message stream that the customer wants to onboard into Splunk. Examples: Windows Security logs, Linux /var/log/secure, firewall syslog, AWS CloudTrail, application JSON logs, O365 audit logs, or HEC application events.

What it is not

A data source is not the Splunk server itself. The Splunk server, UF, HF, search head, or indexer is a component. The actual business telemetry moving through it is the data source.

NameA friendly business label for the onboarding request. Example: Windows Security Logs or Palo Alto Firewall Syslog.
MethodHow Splunk collects the data. Example: windows_event, uf, syslog, hec, aws_s3, or api.
Collection TierThe first Splunk layer that touches the data. Example: universal_forwarder, heavy_forwarder, indexer, or external service.
Target Index and SourcetypeIndex is where the data is stored. Sourcetype is the label Splunk uses to identify the format and apply parsing rules.
How to fill one row
  • Use one row per onboarding pattern, not one row per individual file or host. Add production status, owner, last data seen, and expected health state for daily operations.
  • Describe where the data comes from in Source Location, such as a log path, Windows channel, syslog device type, API name, bucket, queue, or topic.
  • Set App / TA to the add-on or app you expect to deploy for collection or parsing.
Starter example

Windows Security Logs
Category = security
Method = windows_event
Source Location = Windows Event Log: Security
Collection Tier = universal_forwarder
Target Index = winevent

Troubleshooting workbench

This section is environment-aware and issue-aware. It reads the current component inventory, the data-source onboarding matrix, and the clustering toggles, then tells you which component to access first and which Splunk commands to run for that exact failure pattern.

How to use this section
Fastest path: choose Issue based triage, pick an Issue group, then optionally narrow to a host like uf1, idx2, cm1, or sh1. Use Environment runbooks when you already know which host you want to log into first. The command boxes tell you exactly where to run each command.
